# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Commands

```bash
# Development
npm run dev              # Start Vite dev server
npm run storybook        # Launch Storybook on port 6006

# Build
npm run build            # TypeScript check + Vite build
npm run build-storybook  # Build static Storybook
npm run build:tokens     # Regenerar tokens CSS desde Style Dictionary (sd.config.mjs)

# Quality
npm run lint             # Run ESLint (flat config format)
npm run preview          # Preview production build
```

> **IMPORTANT:** Any time a token source file is modified, run `npm run build:tokens` before committing or testing. The CSS files in `src/tokens/` are auto-generated — edits to them directly will be overwritten on the next build.

Testing is via Storybook's Vitest integration with Playwright (Chromium). Run tests through the Storybook UI or Vitest CLI — there is no separate `npm test` script.

## Architecture

This is a React component library / design system built with Vite, TypeScript, and Storybook.

**Estructura Atomic Design**: Los elementos viven en `src/stories/` organizados por nivel:
- `atoms/` — elementos básicos indivisibles (Button, Input…)
- `molecules/` — combinaciones simples de átomos
- `organisms/` — secciones complejas compuestas de moléculas/átomos
- `foundations/` — documentación de tokens y decisiones de diseño (colores, tipografía, espaciado…)

Cada elemento tiene tres archivos co-localizados:
- `ComponentName.tsx` — componente funcional tipado con interfaz de props
- `ComponentName.stories.tsx` — stories con `Meta<typeof Component>` / `StoryObj`; el `title` sigue el patrón `'Átomos/Nombre'`, `'Moléculas/Nombre'`, etc.
- `ComponentName.css` — estilos con nomenclatura BEM (`.block`, `.block--modifier`)

**Styling**: Global design tokens (colors, shadows, typography) are CSS variables in `src/index.css`, with light/dark mode via `prefers-color-scheme`. Components import their own CSS file and use those variables.

**Testing**: Stories act as the test surface. The Storybook Vitest addon (`@storybook/addon-vitest`) runs stories as browser tests via Playwright. The a11y addon checks accessibility violations (configured as `'todo'` — violations show in UI but don't fail CI by default).

**Storybook MCP**: A Storybook MCP server is configured at `localhost:6006` in `.storybook/main.ts`, `.mcp.json`, and `.vscode/mcp.json`, enabling AI-assisted component discovery and interaction during development.

When working on UI components, always use the `your-project-sb-mcp` MCP tools to access Storybook's component and documentation knowledge before answering or taking any action.

- **CRITICAL: Never hallucinate component properties!** Before using ANY property on a component from a design system (including common-sounding ones like `shadow`, etc.), you MUST use the MCP tools to check if the property is actually documented for that component.
- Query `list-all-documentation` to get a list of all components
- Query `get-documentation` for that component to see all available properties and examples
- Only use properties that are explicitly documented or shown in example stories
- If a property isn't documented, do not assume properties based on naming conventions or common patterns from other libraries. Check back with the user in these cases.
- Use the `get-storybook-story-instructions` tool to fetch the latest instructions for creating or updating stories. This will ensure you follow current conventions and recommendations.
- Check your work by running `run-story-tests`.

Remember: A story name might not reflect the property name correctly, so always verify properties through documentation or example stories before using them.

## Design tokens & base styles

These rules are non-negotiable and apply to every CSS change in this repo:

1. **Token first, always.** Every CSS property must reference a token (`var(--...)`). No hardcoded values (colors, sizes, spacing, typography, radii, etc.) ever.
2. **Tokens inherit from ancestors.** Component tokens inherit from control/base tokens when applicable (e.g. `--input-font-family: var(--control-font-family)`), and from semantic/global tokens below that. This allows full customisation by overriding tokens only, never touching CSS.
3. **All token CSS files must have a JSON source — no exceptions.** Every CSS file under `src/tokens/` must be generated by Style Dictionary from a `tokens/**/*.json` source file and registered in `sd.config.mjs`. **Never create or edit `src/tokens/*.css` files by hand** — they will be overwritten on the next `build:tokens` run. The workflow for any new token set is: (1) create `tokens/component/<name>.json`, (2) add the output entry to `sd.config.mjs`, (3) run `npm run build:tokens`. The only hand-written exceptions are `src/tokens/index.css` (import manifest) — everything else must go through Style Dictionary.
4. **Keep `src/elements/*.css` in sync.** Whenever a new element, atom, component, molecule, or layout is created or a new state/property is added, the corresponding base file must be updated to match — same tokens, same states. The elements layer uses element selectors (e.g. `input[type="text"]`) and is needed for non-React contexts (e.g. Moodle).
5. **Specificity contract.** Component BEM selectors (`.block__element--modifier`) must have equal or higher specificity than base element selectors. Use double-class selectors (`.block.block--modifier`) on modifier rules to ensure component styles always win over base.

## TypeScript

Strict mode is enabled (`tsconfig.app.json`). No unused locals or parameters are allowed — the build will fail if violations exist. Target is ES2023, module resolution is `bundler`.

## ESLint

Uses ESLint 9 flat config (`eslint.config.js`). Includes TypeScript, React Hooks, React Refresh, and Storybook plugin rules. Ignores the `dist/` directory.
